# 05040038 智能评价系统接入指引

本文档给游戏开发侧说明如何接入 `evaluation/main/src` 下的智能评价系统。当前系统由四个子模块组成：

| 模块 | 职责 | 游戏侧是否直接调用 |
| :--- | :--- | :--- |
| `rag_builder.py` | 把医生资料 PDF 构建为本地 ChromaDB RAG 知识库 | 资料库初始化/更新时调用 |
| `evaluator.py` | 本地规则评分，生成玩家得分 JSON 和可视化图 | 通常不直接调用 |
| `ai_processor.py` | 读取评分结果和 RAG，调用 LLM 生成教学反馈 JSON | 通常不直接调用 |
| `main.py` | 统一入口：先评分，再生成 AI 教学反馈 | 游戏流程主要调用 |

`evaluation/test/src/build_rag.py` 和 `evaluation/test/src/test_evaluator.py` 是完整调用示例。游戏侧接入时应参考它们的流程，但不要把测试路径和测试密钥写死进游戏。

---

## 1. 总体接入流程

游戏侧分两条链路：

1. **资料库构建链路**：首次启动、医生资料更新、RAG 数据库不存在时执行。
2. **单局评测链路**：每次玩家完成手术模拟后执行。

```text
医生资料 PDF
  -> rag_builder.py
  -> ChromaDB RAG 知识库

病例信息 + GT 标注 + 玩家日志 + 评分细则 + 提示词 + RAG
  -> main.py
  -> evaluator.py 本地评分
  -> ai_processor.py AI 教学反馈
  -> score_result.json + teaching_feedback.json
```

游戏侧建议只暴露两个上层动作：

```text
build_or_update_rag()
evaluate_finished_session()
```

## 3. 输入文件约定

一次完整评测至少需要以下文件：

| 输入 | 示例 | 说明 |
| :--- | :--- | :--- |
| 病例文本 JSON | `3976863_introduction.json` | 病例描述、诊断等文本信息，供 AI 反馈使用 |
| GT 区域 JSON | `3976863_after.json` | LabelMe 标注，包含 `[区域编号]_gt_outer` 和可选 `[区域编号]_gt_inner_[序号]` |
| GT 参数 CSV | `3976863_after.csv` | 与 GT JSON 同名同目录，记录每个区域的专家参数 |
| 玩家日志 JSON | `3976863_player.json` | 玩家有效光斑、参数、坐标、`radius_px` |
| 评分细则 Markdown | `评分细则v5.md` | 同时给人和 AI 使用 |
| AI 提示词 | `llm_structured_scoring_report_prompt.txt` | 控制教学反馈输出风格 |
| RAG 数据库目录 | `evaluation/test/rag/chroma_db` | ChromaDB 持久化目录 |
| 评分配置 JSON | `config.json` 或运行时生成配置 | 控制评分阈值、玩家区域重建参数等 |

### 3.1 GT 区域 JSON

GT 使用 LabelMe 格式。当前支持区域级标注：

```text
[区域编号]_gt_outer
[区域编号]_gt_inner_[内圈序号]
```

示例：

```json
{
  "shapes": [
    {
      "label": "1_gt_outer",
      "points": [[...], [...]]
    },
    {
      "label": "1_gt_inner_1",
      "points": [[...], [...]]
    }
  ]
}
```

规则：

- 一个区域只能有一个 outer。
- 一个区域可以有多个 inner。
- outer 和 inner 的区域编号必须能在 CSV 中找到。
- JSON 中有区域但 CSV 缺参数，直接报错。
- CSV 中有区域但 JSON 没标注，直接报错。

### 3.2 GT 参数 CSV

与 GT JSON 同名同目录。例如：

```text
3976863_after.json
3976863_after.csv
```

CSV 字段：

```csv
id,power,spot_diameter,exposure,wavelength
1,180,200,200,532
```

字段映射：

| CSV 字段 | 内部字段 |
| :--- | :--- |
| `power` | `power` |
| `spot_diameter` | `spot_size` |
| `exposure` | `exposure_time` |
| `wavelength` | `wavelength` |

### 3.3 玩家日志 JSON

玩家日志必须包含 `shots` 数组。每个有效光斑必须包含：

```json
{
  "id": 1,
  "pos": [1000.0, 1200.0],
  "radius_px": 31.8,
  "is_trial": false,
  "params": {
    "power": 180.0,
    "spot_size": 200.0,
    "exposure_time": 200.0,
    "wavelength": 532.0
  }
}
```

强约束：

- `pos` 是图像坐标。
- `radius_px` 是像素半径，不是直径。
- `radius_px <= 0` 直接报错。
- 有效光斑缺少 `radius_px` 直接报错。
- `is_trial=true` 的试打点不参与主要评分。
- 游戏侧必须保证 `player.png`、`before.png`、`after.png` 与 GT 标注图使用同一图像尺寸、同一坐标系。
- 玩家日志中的 `pos` 和 `radius_px` 必须已经对应上述统一坐标系。
- 正式接入时不允许依赖坐标缩放配置修正玩家日志。图像尺寸或坐标系不一致时，应在游戏侧数据导出阶段修正后再调用评分模块。

## 4. RAG 知识库构建

参考文件：

```text
evaluation/test/src/build_rag.py
```

游戏侧在以下场景调用 RAG 构建：

- 第一次安装/第一次启动，还没有 ChromaDB。
- 医生新增、删除、替换 PDF 资料。
- embedding 模型更换。

主接口：

```python
from rag_builder import build_or_update_rag_database

status, msg, stats = build_or_update_rag_database(
    raw_docs_dir="evaluation/test/rag/raw_docs",
    chroma_db_path="evaluation/test/rag/chroma_db",
    manifest_path="evaluation/test/rag/rag_manifest.json",
    parsed_cache_dir="evaluation/test/rag/parsed_cache",
    chunk_cache_dir="evaluation/test/rag/chunk_cache",
    llm_config=llm_config,
    embedding_config=embedding_config,
)
```

返回：

```python
tuple[int, str, dict | None]
```

成功：

```python
status == 1
msg == "success"
stats = {
  "indexed_files": 1,
  "skipped_files": 0,
  "deleted_files": 0,
  "chunk_count": 123,
  "collection": "clinical_knowledge",
  "chroma_db_path": "...",
  "manifest_path": "..."
}
```

失败：

```python
status == 0
msg 包含 traceback
stats is None
```

### 4.1 RAG 目录结构建议

游戏侧可以参考测试目录：

```text
evaluation/test/rag/
  raw_docs/        # 医生提供的 PDF 原始资料
  parsed_cache/    # PDF 解析页缓存
  chunk_cache/     # chunk 缓存
  chroma_db/       # ChromaDB 持久化向量库
  rag_manifest.json
```

`rag_builder.py` 会：

- 扫描 `raw_docs_dir` 下的 PDF。
- 解析 PDF 文本。
- 切分 chunk。
- 调用 embedding 接口。
- 写入 ChromaDB collection：`clinical_knowledge`。
- 用 manifest 记录文件 hash，未变化的 PDF 会跳过。
- raw_docs 删除的 PDF，会从向量库删除对应 chunk。

### 4.2 embedding 配置

OpenAI-compatible embedding：

```json
{
  "provider": "openai_compatible",
  "base_url": "https://api.example.com/v1",
  "api_key": "xxx",
  "model": "text-embedding-3-small",
  "timeout_seconds": 60
}
```

本地测试可用 hash embedding：

```json
{
  "provider": "local_hash"
}
```

正式接入建议使用真实 embedding 模型。`local_hash` 只适合无外网自检，不适合正式教学反馈。

## 5. 单局评测统一入口

游戏侧主要调用：

```text
evaluation/main/src/main.py
```

主接口：

```python
from main import evaluate_with_ai_feedback

status_code, msg, scoring_data, feedback_data = evaluate_with_ai_feedback(
    case_info_json_path=case_info_json_path,
    player_json_path=player_json_path,
    question_json_path=question_json_path,
    rubric_md_path=rubric_md_path,
    prompt_path=prompt_path,
    chroma_db_path=chroma_db_path,
    scoring_config_json_path=scoring_config_json_path,
    scoring_output_json_path=scoring_output_json_path,
    teaching_feedback_output_json_path=teaching_feedback_output_json_path,
    llm_config=llm_config,
    embedding_config=embedding_config,
    param_tolerance_ratio={
        "power": 0.25,
        "spot_size": 0.25,
        "exposure_time": 0.25
    },
)
```

返回：

```python
tuple[int, str, dict | None, dict | None]
```

| 返回值 | 含义 |
| :--- | :--- |
| `status_code` | `1` 成功，`0` 失败 |
| `msg` | 成功为 `success`，失败时包含原因 |
| `scoring_data` | 本地评分 JSON 对象 |
| `feedback_data` | AI 教学反馈 JSON 对象 |

### 5.1 游戏层如何拿到全部可视化图

游戏层拿全部可视化图要看两处：

- `scoring_data["debug_visualization_paths"]`：overlay 和 crop 图
- `scoring_data["dimensions"]["dim2_energy"]["visualization"]`：heatmap 和 wavelength map

`evaluate_with_ai_feedback(...)` 返回成功后：

```python
status_code, msg, scoring_data, feedback_data = evaluate_with_ai_feedback(...)

if scoring_data is None:
    raise RuntimeError(msg)

overlay_paths = scoring_data["debug_visualization_paths"]
heatmap_paths = scoring_data["dimensions"]["dim2_energy"]["visualization"]

player_overlay_path = overlay_paths["player_only"]
gt_overlay_path = overlay_paths["gt_only"]
combined_overlay_path = overlay_paths["combined"]

player_overlay_crop_path = overlay_paths["player_only_crop"]
gt_overlay_crop_path = overlay_paths["gt_only_crop"]
combined_overlay_crop_path = overlay_paths["combined_crop"]
before_crop_path = overlay_paths["before_crop"]
after_crop_path = overlay_paths["after_crop"]

gt_power_heatmap_path = heatmap_paths["gt_power_heatmap"]
player_power_heatmap_path = heatmap_paths["player_power_heatmap"]
error_power_heatmap_path = heatmap_paths["error_power_heatmap"]
total_error_heatmap_path = heatmap_paths["total_error_heatmap"]
```

如果游戏层希望一次性拿到所有可视化资源，直接合并这两个字典：

```python
all_visualizations = {
    **scoring_data["debug_visualization_paths"],
    **scoring_data["dimensions"]["dim2_energy"]["visualization"],
}
```

当前 overlay/crop 图在 `debug_visualization_paths` 中固定包含 8 个 key：

| key | 文件内容 | 用途 |
| :--- | :--- | :--- |
| `player_only` | 玩家治疗区域 + 玩家点位整图 overlay | 展示玩家实际操作范围 |
| `gt_only` | GT 标准治疗区域整图 overlay | 展示专家标准区域 |
| `combined` | 玩家区域 + GT 区域整图 overlay | 做整图对比 |
| `player_only_crop` | 玩家 overlay 的统一裁剪图 | 聚焦手术区域局部 |
| `gt_only_crop` | GT overlay 的统一裁剪图 | 聚焦标准区域局部 |
| `combined_crop` | combined overlay 的统一裁剪图 | 聚焦局部对比 |
| `before_crop` | 术前原图裁剪图，无标记 | 作为局部背景原图 |
| `after_crop` | 术后原图裁剪图，无标记 | 作为局部背景原图 |

当前 heatmap 图在 `dimensions.dim2_energy.visualization` 中固定包含 13 个 key：

| key | 文件内容 |
| :--- | :--- |
| `gt_power_heatmap` | GT 功率热力图 |
| `player_power_heatmap` | 玩家功率热力图 |
| `error_power_heatmap` | 功率误差热力图 |
| `gt_spot_size_heatmap` | GT 光斑大小热力图 |
| `player_spot_size_heatmap` | 玩家光斑大小热力图 |
| `error_spot_size_heatmap` | 光斑大小误差热力图 |
| `gt_exposure_time_heatmap` | GT 曝光时间热力图 |
| `player_exposure_time_heatmap` | 玩家曝光时间热力图 |
| `error_exposure_time_heatmap` | 曝光时间误差热力图 |
| `gt_wavelength_map` | GT 波长分布图 |
| `player_wavelength_map` | 玩家波长分布图 |
| `error_wavelength_map` | 波长误差图 |
| `total_error_heatmap` | 维度二总误差热力图 |

说明：

- 这 21 个路径都是评估模块已经生成并落盘后的绝对路径。
- 5 张 crop 图使用同一个裁剪框，尺寸完全一致，游戏侧可以直接做切换显示或并排对比。
- heatmap 路径位于 `dimensions.dim2_energy.visualization`，它们和评分结果是一一对应的，游戏侧不需要自己拼接文件名。
- `debug_visualization_path` 只是兼容字段，始终等于 `debug_visualization_paths["combined"]`。如果要拿全量可视化，必须读取 `debug_visualization_paths`。
- 如果只想拿“全部可视化资源”，推荐直接合并 `debug_visualization_paths` 和 `dimensions.dim2_energy.visualization`，不要依赖单个兼容字段。

### 5.2 失败策略

`main.py` 的策略是：

- 本地评分失败：返回 `status_code=0`，`scoring_data=None`，`feedback_data=None`。
- 本地评分成功但 AI 失败：返回 `status_code=0`，保留 `scoring_data`，`feedback_data=None`。
- 全部成功：返回 `status_code=1`，同时返回 `scoring_data` 和 `feedback_data`。

游戏侧应按这个逻辑处理：

```python
if status_code == 1:
    show_score(scoring_data)
    show_feedback(feedback_data)
    show_visualizations({
        **scoring_data["debug_visualization_paths"],
        **scoring_data["dimensions"]["dim2_energy"]["visualization"],
    })
elif scoring_data is not None:
    show_score(scoring_data)
    show_visualizations({
        **scoring_data["debug_visualization_paths"],
        **scoring_data["dimensions"]["dim2_energy"]["visualization"],
    })
    show_message("AI 教学反馈生成失败，可稍后重试")
else:
    show_message("评分失败，需要检查输入数据")
```

## 6. 评分配置

评分配置由游戏侧传入。测试侧做法见：

```text
evaluation/test/src/test_evaluator.py
```

当前推荐运行时配置：

```json
{
  "scoring_policy": {
    "param_tolerance_ratio": {
      "power": 0.25,
      "spot_size": 0.25,
      "exposure_time": 0.25
    },
    "spatial_parameter_field": {
      "field_resolution": "source_image_pixels",
      "field_sigma_px": 25.0,
      "heatmap_crop_padding_px": 40
    },
    "player_area_geometry": {
      "mode": "clustered_disk_concave_envelope",
      "cluster_distance_ratio": 3.5,
      "concave_hull_ratio": 0.3,
      "perimeter_sample_count": 16,
      "min_envelope_shots": 3
    },
    "penalty_rules": {
      "points_per_overlap": 1.0,
      "overlap_min_depth_ratio": 0.15
    }
  }
}
```

### 6.1 `param_tolerance_ratio`

连续参数容忍比例：

```text
t_k(x,y) = GT_k(x,y) * ratio_k
```

当前默认：

```json
{
  "power": 0.25,
  "spot_size": 0.25,
  "exposure_time": 0.25
}
```

波长不使用容忍比例，按离散一致性评分。

### 6.2 `player_area_geometry`

玩家治疗区域重建配置：

```json
{
  "mode": "clustered_disk_concave_envelope",
  "cluster_distance_ratio": 3.5,
  "concave_hull_ratio": 0.3,
  "perimeter_sample_count": 16,
  "min_envelope_shots": 3
}
```

含义：

| 字段 | 含义 |
| :--- | :--- |
| `cluster_distance_ratio` | 用 `median(radius_px) * ratio` 作为治疗簇聚类距离 |
| `concave_hull_ratio` | 每个治疗簇外缘点生成 concave hull 的比例参数 |
| `perimeter_sample_count` | 每个光斑圆周采样点数量 |
| `min_envelope_shots` | 小于该数量的簇不生成包络，保留原始圆盘 |

此算法的目标是模拟医生手工沿光斑外缘/内缘画治疗区域的方式，同时支持包绕、片状、条状、多区域。

### 6.3 `penalty_rules`

重叠扣分配置：

```json
{
  "points_per_overlap": 1.0,
  "overlap_min_depth_ratio": 0.15
}
```

对任意两个光斑，系统先计算重叠深度：

```text
overlap_depth = r1 + r2 - center_distance
```

再计算归一化重叠深度：

```text
overlap_depth_ratio = overlap_depth / min(r1, r2)
```

只有 `overlap_depth_ratio >= 0.15` 才算一次有效重叠并扣分。边缘轻微接触不扣分。

## 9. 四个主模块的关系

### 9.1 `rag_builder.py`

只负责 RAG 构建，不参与单局评分。

主函数：

```python
build_or_update_rag_database(...)
```

输入 PDF，输出 ChromaDB 和 manifest。

### 9.2 `evaluator.py`

只负责本地规则评分。

主函数：

```python
evaluate(
    question_json_path,
    player_json_path,
    scoring_output_json_path,
    config_json_path=None,
    param_tolerance_ratio=None,
)
```

游戏侧一般不直接调用它，除非只想做本地评分、不需要 AI 反馈。

### 9.3 `ai_processor.py`

只负责 AI 教学反馈。

主函数：

```python
generate_teaching_feedback_report(...)
```

它不会重新评分，不会改分，只解释 `score_result.json`。

### 9.4 `main.py`

统一入口，游戏侧优先调用。

主函数：

```python
evaluate_with_ai_feedback(...)
```

内部流程固定：

```text
校验输入路径
-> evaluator.evaluate()
-> 读取 score_result.json
-> 读取病例、评分细则、提示词
-> ai_processor.generate_teaching_feedback_report()
-> 写 teaching_feedback.json
-> 返回 scoring_data + feedback_data
```

## 11. 常见问题

### Q1：游戏侧应该直接调用 `evaluator.py` 还是 `main.py`？

需要 AI 教学反馈时调用 `main.py`。

### Q2：RAG 是否每局都要构建？

不需要。RAG 只在首次启动或资料更新时构建。每局评测只读取已有 ChromaDB。

### Q3：为什么 `main.py` 不自己构建 RAG？

RAG 构建耗时较长，而且只和资料库更新有关，不应该阻塞单局评分流程。游戏侧应在启动阶段或后台任务中构建。

### Q4：AI 反馈失败会不会导致本地评分丢失？

不会。`main.py` 在本地评分成功后会先写出 `score_result.json`。AI 失败时仍返回 `scoring_data`。

### Q5：玩家治疗区域为什么不是直接用光斑圆盘并集？

GT 区域是医生沿治疗区域外缘和内缘手工标注的连续区域，不是每个光斑圆盘的机械并集。当前玩家区域重建使用光斑半径生成治疗簇包络，使语义更接近 GT 标注方式，同时保留孤立小簇，不让其改变主区域轮廓。
